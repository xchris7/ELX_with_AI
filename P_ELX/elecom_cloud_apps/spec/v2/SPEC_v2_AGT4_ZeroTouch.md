# AdminLink Requirements Specification

## Client Request

**EN:** Client wants to use the AdminLink service.  

---

## Glossary

| Term | Definition |
|------|-----------|
| AdminLink | The cloud-based device management service |
| Agent | The on-device software component that communicates with AdminLink |
| Device ID | UUID v4 generated by the device; used as identifier in all API calls |
| Registration Code | A 9-digit hex code generated for device registration (`register code` / `regist_cd`) |
| dev_id_changed | Flag (0 or 1) in API response; 1 indicates the device has been factory-reset after prior registration |
| agt_daily_sec | Seconds offset from 22:00:00 for daily processing execution time |
| agt_upload_sec | Seconds offset from :00:00 for hourly status upload timing |
| tmp_reg_expiry | Temporary registration expiry datetime (Zero-touch flow only) |
| re-registerable | State where a device was previously registered, factory-reset, and can be re-registered |
| Periodic Notification | `sts_type = 0` — status update sent by scheduled hourly process |
| Event Notification | `sts_type = 1` — status update triggered by an event |
| Optional Notification | `sts_type = 2` — status update triggered manually |

## Device Registration State Machine

| State | Description | Triggered By | Transitions To |
|-------|-------------|-------------|----------------|
| `Disabled` | AdminLink function is turned off | User disables via Web UI | `Unregistered` (when re-enabled) |
| `Unregistered` | Not registered to AdminLink service | API returns 401; factory reset | `Registered`, `Checking` |
| `Registered` | Successfully registered | API returns 201 or 200 (dev_id_changed=0) | `Unregistered`, `Checking`, `re-registerable` |
| `Checking` | Registration status cannot be determined | API returns other error; no response | `Registered`, `Unregistered` (after retry) |
| `re-registerable` | Factory-reset after registration | API returns 200 (dev_id_changed=1) | `Registered` (after re-registration) |

# AGT.4 — Device registration and configuration shall be completable through operations on the AdminLink server side.

**Rationale:** We want to complete the setup process by simply installing the device on site.

Subjects are switches and APs.

## AGT.4.1 — Automatic Registration Flow Execution Judgment

**Rationale:** To support conventional registration methods as well as automatic temporary registration for zero-touch setting, we would like to determine the device status to determine if the device is ready for zero-touch setting at the start of device operation.

**Description:** Judges whether or not there is a change in the device settings, and executes the automatic registration flow if there is a change, or the normal startup flow if there is no change.

### AGT.4.1.0 — Shipping Configuration

#### `AGT.4.1.1`

When shipped from the factory, the device shall have the following settings
- DHCP client functionality must be enabled.
- AdminLink functionality must be enabled.
- **"Remote Control Permission"** in **"AdminLink Advanced"** must be **"Enabled"**.
- MAC address" and **"resist code"** shall be written in the non-volatile memory area.
- Time synchronization by NTP/SNTP must be selected (time zone: UTC+09:00 / daylight saving time: OFF)

### AGT.4.1.10 — Processing at device startup (whether cold start or hot start)

#### `AGT.4.1.11`

When the device is powered on or restarted, if the normal startup flow is completed and all of the following conditions are met, <Configuration Change Existence Judgment> is executed.
- The device must have completed time synchronization via NTP/SNTP.
- The device must be able to communicate with the AdminLink server
- The device shall be ready to process "Change Settings" (Remote ID: 5070) and "Restore Settings" (Remote ID: 5080) requests.

### AGT.4.1.20 — Determination of whether or not settings have been changed

#### `AGT.4.1.21`

If the **"Administrator Password"** has not been changed from the default, the **"Automatic Registration Flow"** shall be executed.

#### `AGT.4.1.22`

If there is a difference in settings, the process is terminated.

## AGT.4.2 — Automatic Registration Flow

### AGT.4.2.0 — Calling Device Registration API

#### `AGT.4.2.1`

A new device ID shall be generated and retained in RAM.
The device ID shall be in UUID Ver4 format (AGT.1.4.2).
The generated device ID should NOT be written to non-volatile memory at this time.

#### `AGT.4.2.2`

The **"Device Registration API"** shall be called with the generated device ID, the resist code held by the device, and the MAC address as parameters.
> **Note:** For details on the "Device Registration API," refer to the "WebAPI Specification.
The request body must include the following keys:
- Product category (`prdct`)
- Register code (`regist_cd`)
- Remote operation permission (`remote_flg`) — set to "Allowed (1)".
- Proxy setting (`proxy_flg`) — set to "Disabled (0)".
- Device ID (`dev_id`)
- Product series name (`prdct_series`)
- Product model name (`prdct_name`)
- Product model identifier string (`prdct_id`)
- Primary IP address (`ip_adr`)
- Primary MAC address (`mac_adr`)
- Management software mnemonic (`ms_mnemonic`)
- Management software version (`ms_ver`)
- Agent mnemonic (`agt_mnemonic`)
- Agent version (`agt_ver`)

For the value of each key (e.g., product category, product series name, etc.),
Refer to the Appendix "AdminLink_DeviceRegistrationAPI_Parameters(APSW)".

#### `AGT.4.2.3`

The response status of the **"Device Registration API"** shall be obtained and processing shall be executed according to the "Status Code".

### AGT.4.2.10 — Device Registration API status code is 201 (Created)

#### `AGT.4.2.11`

The success of the **"Device Registration API"** shall be recorded in the log.
In addition, the values of the following parameters returned in the response body shall also be recorded in the log:

- Device ID change flag (`dev_id_changed`)
- Seconds for upload timing calculation (`agt_upload_sec`)
- Seconds for daily processing execution time calculation (`agt_daily_sec`)
- Temporary registration expiry datetime (`tmp_reg_expiry`) (If included in the response body.)

#### `AGT.4.2.12`

The values of parameters returned in the response body shall be retained in RAM.
This process shall NOT write to non-volatile memory.
The keys to be held in RAM are as follows:
- Device ID (`dev_id`)
- Seconds for daily processing execution time calculation (`agt_daily_sec`)
- Seconds for upload timing calculation (`agt_upload_sec`)
- Temporary registration expiry datetime (`tmp_reg_expiry`) (If included in the response body.)

NOTE: If the **"tmp_reg_expiry"** key is not included in the response body, the **"tmp_reg_expiry"** value must NOT be retained in RAM.

#### `AGT.4.2.13`

If the **"tmp_reg_expiry"** key is included in the response body, **"Zero touch wait processing (AGT.4.3)"** shall be executed.

#### `AGT.4.2.14`

If the **"tmp_reg_expiry"** key is not included in the response body, the following registration information contained in the response body shall be stored in non-volatile memory and the <End of automatic registration flow> (AGT.4.2.40) process shall be executed.
The registration information to be stored:
- Device ID (`dev_id`)
- Seconds for daily processing execution time calculation (`agt_daily_sec`)
- Seconds for upload timing calculation (`agt_upload_sec`)
Note: These parameters are stored in the non-volatile memory of the device itself, but must not be written out to the device's configuration files (cfg or bin).

### AGT.4.2.20 — Device Registration API status code other than 201 (Created)

#### `AGT.4.2.21`

The failure of the **"Device Registration API"** shall be logged.

#### `AGT.4.2.22`

### AGT.4.2.30 — Failure to obtain device registration API status code (no response from WebAPI server)

#### `AGT.4.2.31`

The failure to obtain a response from the **"Device Registration API"** shall be recorded in the log.
Only the first time shall be recorded in the log, and execution by retry shall not be recorded in the log.

#### `AGT.4.2.32`

Wait for 10 seconds, and retry from <Determining whether or not the setting has been changed> (AGT.4.1.20).

### AGT.4.2.40 — Terminate the automatic registration flow

#### `AGT.4.2.41`

The automatic registration flow shall be terminated and processing shall be continued from AGT.2.1.0 <Start agent function at device startup>.

## AGT.4.3 — Zero touch standby process

**Rationale:** I want to change device settings remotely from the server.

**Description:** Starts accepting remote control and changes device settings when a remote control configuration change request is received.

### AGT.4.3.0 — Start of Remote Operation Acceptance

#### `AGT.4.3.1`

To start the resident process of remote control reception (AGT.3.1.1).
For details on remote control reception,
Refer to the "7. Remote control reception flow" sheet in "EJ02.(AdminLink) 01. WebAPI Specification Supplement (Agent_Cloud Linkage Flow).xlsx".

### AGT.4.3.10 — Successful start of remote acceptance

#### `AGT.4.3.11`

The successful start of remote operation reception shall be recorded in the log (AGT.3.1.11).

#### `AGT.4.3.12`

When a remote control request is received from the server, <Receive Remote Control> (AGT.4.3.60) shall be executed.

#### `AGT.4.3.13`

### AGT.4.3.20 — Failure to start remote operation reception

#### `AGT.4.3.21`

If a failure to start remote operation reception occurs, it shall be logged only for the first time, and shall not be logged if a failure occurs in a retry.

#### `AGT.4.3.22`

If the **"tmp_reg_expiry"** value is held in RAM and the current date/time exceeds the **"temporary registration expiration date/time (tmp_reg_expiry)"**, logging and <Zero touch wait process end> must be executed.

#### `AGT.4.3.23`

If the **"tmp_reg_expiry"** value is not kept in RAM, or the current date/time does not exceed the **"temporary registration expiration date/time (tmp_reg_expiry)"**, it must wait for 10 seconds and then retry from <Start remote operation reception>.

### AGT.4.3.30 — Determination of whether or not settings have been changed

#### `AGT.4.3.31`

If the **"Administrator Password"** has not been changed from the default, the **"Automatic Registration Flow"** shall be executed.
The AdminLink function shall remain available for remote operation reception.

#### `AGT.4.3.32`

When there is a change in the setting, the process shall be stopped and <Zero touch standby process end> shall be executed.

### AGT.4.3.40 — Determination of temporary registration expiration date

#### `AGT.4.3.41`

If the value of **"tmp_reg_expiry"** is maintained in RAM and the current date/time exceeds the **"temporary registration expiration date/time (tmp_reg_expiry)"**, logging must be performed and <Zero touch wait process end> must be executed.

#### `AGT.4.3.42`

If the **"tmp_reg_expiry"** value is NOT kept in RAM or the current date/time does NOT exceed the **"temporary registration expiration date/time (tmp_reg_expiry)"**, the system must remain ready to accept remote operations.

### AGT.4.3.50 — Detect that acceptance of remote-control requests has been disconnected.

#### `AGT.4.3.51`

When a disconnection of the communication channel for remote-control reception is detected, the event shall be recorded in the log.

#### `AGT.4.3.52`

After waiting for 10 seconds, retry to reconnect from <Temporary Registration Expiration Date Determination>.

### AGT.4.3.60 — Remote Receiving

#### `AGT.4.3.61`

When the following remote operation requests are received, the process indicated by the received remote operation ID shall be executed.

#### `AGT.4.3.62`

Remote operation ID: If other than 5070 or 5080, <Remote operation execution failure> shall be executed.

### AGT.4.3.70 — Successful execution of remote control

#### `AGT.4.3.71`

If the remote control process is successful, <AdminLink Full Registration Process> shall be executed.

### AGT.4.3.80 — Failure to execute remote control

#### `AGT.4.3.81`

#### `AGT.4.3.82`

Remote operation acceptance must continue to be available.

### AGT.4.3.90 — AdminLink Full Registration Process

#### `AGT.4.3.91`

The following registration information held in RAM shall be stored in non-volatile memory and the registration status shall be set to **"registered"**.
- Device ID (`dev_id`)
- Seconds for daily processing execution time calculation (`agt_daily_sec`)
- Seconds for upload timing calculation (`agt_upload_sec`)
Note: These parameters are stored in the non-volatile memory of the device itself, but must not be written out to the device's configuration files (cfg or bin).

#### `AGT.4.3.92`

The "Zero touch standby process end" shall be executed.

#### `AGT.4.3.93`

Resident processing of remote control reception must continue.

#### `AGT.4.3.94`

Log the completion of this registration of the device.

#### `AGT.4.3.95`

The following event JSON shall be sent to the AdminLink server.
- Agent initialization (Action ID: 5060)

#### `AGT.4.3.96`

Device should be rebooted.

### AGT.4.3.100 — Zero touch standby process end

#### `AGT.4.3.101`

If a **"temporary registration expiration date/time (tmp_reg_expiry)"** is retained in RAM, it shall be destroyed.

#### `AGT.4.3.102`

If the device is unregistered (which means that the device ID is not stored in non-volatile memory), the resident process of remote control reception must be stopped.

#### `AGT.4.3.103`

The zero-touch standby process shall be terminated and the process shall be continued from AGT.2.1.0 <Start agent function at device startup>.
