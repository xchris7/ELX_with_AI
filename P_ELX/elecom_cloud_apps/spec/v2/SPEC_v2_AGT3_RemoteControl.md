# AdminLink Requirements Specification

## Client Request

**EN:** Client wants to use the AdminLink service.  

---

## Glossary

| Term | Definition |
|------|-----------|
| AdminLink | The cloud-based device management service |
| Agent | The on-device software component that communicates with AdminLink |
| Device ID | UUID v4 generated by the device; used as identifier in all API calls |
| Registration Code | A 9-digit hex code generated for device registration (`register code` / `regist_cd`) |
| dev_id_changed | Flag (0 or 1) in API response; 1 indicates the device has been factory-reset after prior registration |
| agt_daily_sec | Seconds offset from 22:00:00 for daily processing execution time |
| agt_upload_sec | Seconds offset from :00:00 for hourly status upload timing |
| tmp_reg_expiry | Temporary registration expiry datetime (Zero-touch flow only) |
| re-registerable | State where a device was previously registered, factory-reset, and can be re-registered |
| Periodic Notification | `sts_type = 0` — status update sent by scheduled hourly process |
| Event Notification | `sts_type = 1` — status update triggered by an event |
| Optional Notification | `sts_type = 2` — status update triggered manually |

## Device Registration State Machine

| State | Description | Triggered By | Transitions To |
|-------|-------------|-------------|----------------|
| `Disabled` | AdminLink function is turned off | User disables via Web UI | `Unregistered` (when re-enabled) |
| `Unregistered` | Not registered to AdminLink service | API returns 401; factory reset | `Registered`, `Checking` |
| `Registered` | Successfully registered | API returns 201 or 200 (dev_id_changed=0) | `Unregistered`, `Checking`, `re-registerable` |
| `Checking` | Registration status cannot be determined | API returns other error; no response | `Registered`, `Unregistered` (after retry) |
| `re-registerable` | Factory-reset after registration | API returns 200 (dev_id_changed=1) | `Registered` (after re-registration) |

# AGT.3 — Must be able to receive remote control requests from AdminLink servers, execute requests, and reply results.

**Rationale:** Client wants to remote control.

## AGT.3.1 — Must be able to receive remote control requests from the AdminLink server while the AdminLink function is enabled and registered.

**Rationale:** Client wants to receive remote control request.

### AGT.3.1.0 — Start remote control reception

#### `AGT.3.1.1`

When AdminLink agent starts or finish the device registration, if the following condition met, it will be on "remote control reception".
- The device registration status is **"registered"**.
> **Note:** If "Remote Control Permission" is "Disabled", but the above condition met, AdminLink agent will be on "remote control reception".

#### `AGT.3.1.2`

When the proxy setting is enabled, communication is done via the proxy.

#### `AGT.3.1.3`

Start remote control reception, if the device registration status changes from something except **"registered"** to "registered,"

#### `AGT.3.1.4`

Remote control reception will resume immediately, if remote control reception stops due to communication errors, etc.,

### AGT.3.1.10 — Remote control reception successfully starts

#### `AGT.3.1.11`

Log the start of remote control reception.
See the 'Log Message' sheet.

### AGT.3.1.20 — remote control reception fail to start

#### `AGT.3.1.21`

While the AdminLink function is enabled and the device registration status is **"Registered"**, the system will retry until the remote control reception is successful.

#### `AGT.3.1.22`

Logs that the remote control reception failed to start.
See the 'Log Message' sheet.

### AGT.3.1.30 — Remote control reception closed

#### `AGT.3.1.31`

The remote control reception is terminated in the following cases.
- When the AdminLink function changes from enabled to disabled in the Web UI.
- When the device registration is canceled.

### AGT.3.1.40 — Remote control reception successfully closed

#### `AGT.3.1.41`

Log the completion of remote control reception.
See the 'Log Message' sheet.

### AGT.3.1.50 — Remote control reception fails to close

#### `AGT.3.1.51`

Log that remote control reception failed to close.
See the 'Log Message' sheet.

### AGT.3.1.60 — Detects disconnection of remote control reception

#### `AGT.3.1.61`

Log that remote control reception is disconnected.
See the 'Log Message' sheet.

#### `AGT.3.1.62`

Start remote control reception.

### AGT.3.1.70 — Perform remote control

#### `AGT.3.1.71`

Process the received remote control ID,  if "Remote Control Permission" is "Enabled" when receiving a remote control request .

#### `AGT.3.1.72`

Only the received remote operation ID 5010 (Remote Operation Permission) is executed if "Remote Operation Permission" is "Disabled" when receiving a remote control request . Other remote control IDs will not be executed, and an error will be returned to the AdminLink server side for the remote control request.

#### `AGT.3.1.73`

If an unsupported remote control ID is received, an error will be returned to the AdminLink server side for the remote control request.

#### `AGT.3.1.74`

When remote control ID (4020) **"Upload Log File"** is received, it is executed only when the following conditions are met. If the condition is not met, the operation is not executed and an error is returned to the AdminLink server side.
- **"Remote control permission"** is **"enabled"**.
- **"Log file upload permission"** is **"enabled"**.

#### `AGT.3.1.75`

When the remote control ID (4030) "Upload Configuration File" is received, it is executed only when the following conditions are met. If the condition is not met, the function is not executed and an error is returned to the AdminLink server side.
- **"Remote control permission"** is **"enabled"**.
- **"Configuration file upload permission"** is **"enabled"**.

#### `AGT.3.1.76`

When remote control ID (4040) "Upload connection client file" is received, it is executed only if the following conditions are met. If the condition is not met, the operation is not executed and an error is returned to the AdminLink server side.
- **"Remote control permission"** is **"enabled"**.
- **"Connected client file upload permission"** is **"enabled"**.

#### `AGT.3.1.77`

### AGT.3.1.80 — Successful execution of remote control

#### `AGT.3.1.81`

Log the completion of the execution, if the process indicated by the remote control ID is successfully executed.
See the 'Log Message' sheet.

#### `AGT.3.1.82`

Send the following event JSON to the AdminLink server.
Action ID (8010) Remote control execution successfully completed

#### `AGT.3.1.83`

### AGT.3.1.90 — Remote control failed to execute

#### `AGT.3.1.91`

Log the error if the execution of the process indicated by the remote control ID fails.
See the 'Log Message' sheet.

#### `AGT.3.1.92`

Send the following event JSON to the AdminLink server.
Action ID (8020) Remote control execution error

## AGT.3.2 — Functions requested via remote operation (including file downloads) shall be executable.

- Target Remote Operation ID
- Firmware update
- Change Settings
- Restore Settings

#### `AGT.3.2.2`

### AGT.3.2.10 — Web API for obtaining the file upload URL successfully obtained

### AGT.3.2.20 — Web API for obtaining the file upload URL fails to obtain

#### `AGT.3.2.41`

#### `AGT.3.2.81`

## AGT.3.3 — Functions requested via remote operation (including file upload) shall be executable.

**Rationale:** I want to perform remote operations that require file uploads to the server, such as log file uploads and configuration file uploads.

### AGT.3.3.0 — Receiving and Analyzing Remote Control Information

#### `AGT.3.3.1`

Remote operation information received from the server shall be checked to see if it is the target **"remote operation ID"** or not.
Target Remote Operation ID
- Upload log (4020)
- Upload configuration file (4030)
- Upload connection client file (4040)
- Get Settings (5060)
If it is a target remote operation ID, the process shall be executed according to each remote operation ID.
See the "Upload,Download Files" sheet for details on files to be uploaded.

#### `AGT.3.3.2`

When the remote operation ID is **"Upload Log (4020)"**, a **"log file"** is created for upload from the device's RAM log.
Note: If there are zero log messages on the device side, the log file creation shall fail and <File Upload Failure> shall be executed.
This function is executed only when **"Remote control permission"** is **"Enabled"** and **"Log file upload permission"** is **"Enabled"**.
If **"Disabled"**, it will not be executed.

#### `AGT.3.3.3`

If the remote operation ID is **"Configuration File Upload (4030)"**, create a "configuration file" for upload.
This function is executed only when **"Remote control permission"** is **"Enabled"** and **"Configuration file upload permission"** is **"Enabled"**.
If **"Disabled"**, it will not be executed.

#### `AGT.3.3.4`

If the remote operation ID is **"Upload Connection Client File (4040)"**, create a **"connection client file"** for upload.
This function is executed only when **"Remote control permission"** is **"Enabled"** and **"Connected client file upload permission"** is **"Enabled"**.
If **"Disabled"**, it will not be executed.

#### `AGT.3.3.5`

When the remote operation ID is **"Setting acquisition (5060)"**, a **"setting data JSON file"** is created for upload based on the current setting values.
This function is executed only when **"Remote control permission"** is **"Enabled"**.
If **"Disabled"**, it will not be executed.

#### `AGT.3.3.6`

After device Log file creation complete, upload it to the AdminLink server.

#### `AGT.3.3.7`

To upload a file, call **"Get URL for File Upload API"** to get the URL for uploading.

### AGT.3.3.10 — Successfully retrieved upload URL.

#### `AGT.3.3.11`

Upload the file to the URL obtained.

### AGT.3.3.20 — Failed to retrieve upload URL.

#### `AGT.3.3.21`

Log if the Web API to retrieve File Upload URL fails.
See the 'Log Message' sheet.

#### `AGT.3.3.22`

### AGT.3.3.30 — Successful upload file

#### `AGT.3.3.31`

After the upload is complete, call the file upload completion notification Web API to notify the AdminLink server of the completion of the upload.
When a file upload is executed by remote control request, the **"automatic upload flag (auto_flg)"** included in the "File upload completion notification Web API" parameter must be set to 0 (manual).

### AGT.3.3.40 — Failed to upload

#### `AGT.3.3.41`

Log if file fails to uploaded.
See the 'Log Message' sheet.

#### `AGT.3.3.42`

### AGT.3.3.50 — File upload completion notification API successful

#### `AGT.3.3.51`

### AGT.3.3.60 — File upload completion notification Web API failure

#### `AGT.3.3.61`

Log it the execution of the file upload completion notification Web API fails.
See the 'Log Message' sheet.
