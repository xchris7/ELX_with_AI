# SKILL: Device Registration API

## Trigger Conditions

Use this SKILL when the user asks to:
- Call or implement the Device Registration API
- Build or validate a request payload for device registration
- Diagnose an error response from the API
- Verify whether a product serial number format is correct
- Understand what parameters are required under specific conditions
- Generate sample code (Python, JS, curl, etc.) for device registration

---

## API Overview

| Item | Value |
|------|-------|
| URI | `https://api.admin-link.net/v1/devices` |
| Method | `POST` |
| Content-Type | `application/json` |
| Purpose | Register a device to the service (called by the device-side agent) |

---

## Execution Steps

### Step 1 — Identify the Registration Mode

Determine which registration mode applies before building the request:

| Mode | Trigger | Key Rule |
|------|---------|----------|
| **Manual Registration** | User registers via agent's Web UI screen | Use `dev_reg_cd`; do NOT send `regist_cd` |
| **Auto Registration** | Device installs/starts up automatically (AP/SW only) | Use `regist_cd`; do NOT send `dev_reg_cd` |

> ⚠️ `dev_reg_cd` and `regist_cd` are mutually exclusive. Send only one. The key itself must NOT be present in the request if not used.

---

### Step 2 — Build the Request Body

#### Required Fields (always include)

| Key | Type | Max Length | Description |
|-----|------|------------|-------------|
| `prdct` | string | 20 | Product category. Must be one of: `NAS`, `AP`, `SW` |
| `dev_id` | string | 40 | Device ID (GUID), generated by the agent |
| `prdct_id` | string | 50 | Product model number identification string. Validated against device master using product code from serial number |
| `mac_adr` | string | 30 | Representative MAC address of the device |
| `ms_mnemonic` | string | 50 | Management software mnemonic (NAS software category 1 or 2 only) |
| `ms_ver` | string | 20 | Management software version |
| `agt_mnemonic` | string | 50 | Agent mnemonic (NAS software category 3 only) |
| `agt_ver` | string | 20 | Agent version |

#### Conditional Fields (△ — required depending on mode/context)

| Key | Max Length | Condition | Rule |
|-----|------------|-----------|------|
| `dev_reg_cd` | 255 | Manual registration via Web UI | Required. Strip leading/trailing spaces before sending |
| `regist_cd` | 8 | Auto registration (AP/SW only) | Required. Do NOT send in Web UI flow |
| `dev_name` | 100 (half-width) | First registration, manual mode only | Required on first registration. Must NOT be sent when using `regist_cd` |
| `serial_no` | 50 | First registration, manual mode only | Required on first registration. Must NOT be sent when using `regist_cd`. See *Serial Number Format Rules* |
| `remote_flg` | 1 | First registration | `0`=Not permitted, `1`=Permitted. **NextGiga: set to `1` when using `regist_cd`** |
| `proxy_flg` | 1 | First registration | `0`=Disable, `1`=Enable. **NextGiga: set to `0` when using `regist_cd`** |
| `os_cd` | 20 | NAS only | Required for NAS. Only `WIN` or `LINUX` accepted |
| `ng_type` | 3 | NAS only | Required for NAS. Only predefined types accepted |

#### Optional Fields

| Key | Max Length | Description |
|-----|------------|-------------|
| `email` | 255 | Legacy NAS field. Do NOT send — presence triggers Azure-agent error (400/4008) |
| `password` | — | Optional |
| `remarks` | 255 (half-width) | Device remarks. Leading/trailing spaces auto-removed |
| `prdct_series` | 50 | Product series name |
| `prdct_name` | 50 | Product model name |
| `ip_adr` | 15 | Representative IP address |
| `mb_mnemonic` | 50 | Motherboard mnemonic |
| `mb_name` | 100 | Motherboard name |
| `mb_sn` | 50 | Motherboard serial number |
| `bios_ver` | 10 | Motherboard BIOS version |
| `os_mnemonic` | 50 | OS mnemonic |
| `os_name` | 100 | OS name |
| `os_ver` | 100 | OS version |
| `host_name` | 100 | Host name |
| `ms_mnemonic` | 50 | Management software mnemonic |
| `agt_ver` | 20 | Agent version |
| `auto_opt1`–`auto_opt9` | 255 | Reserved for future data collection |

### Step 3 — Validate Serial Number Format (if `serial_no` is used)

#### 12–13 digit format (legacy)

| Digit(s) | Value | Meaning |
|----------|-------|---------|
| 1 | 0–9 | Last digit of manufacturing year (A.D.) |
| 2 | 1–C | Manufacturing month (hexadecimal: 1=Jan ... C=Dec) |
| 3–7 | 1 uppercase letter + 4 digits | Product code |
| 8+ | 5+ alphanumeric characters | Serial number |
| 13 | A–Z (optional) | Product version |

Serial number sub-format:
- If digit 8 is a **number** → digits 8–12 are the 5-digit decimal serial number
- If digit 8 is a **letter** → it indicates product category; serial number is 4-digit decimal

#### 14–15 digit format (new)

| Digit(s) | Value | Meaning |
|----------|-------|---------|
| 1–2 | 00–FF (hex) | Factory code (**New field**) |
| 3 | 0–9 | Last digit of manufacturing year |
| 4 | 1–C | Manufacturing month (hexadecimal) |
| 5–9 | 1 uppercase letter + 4 digits | Product code |
| 10+ | 5+ alphanumeric characters | Serial number |
| 15 | A–Z (optional) | Product version |

> If the serial number contains lowercase letters → auto-convert to uppercase before sending.

**Example:**

| Format | Serial Number |
|--------|---------------|
| Legacy (12-digit) | `01L016300001A` |
| New (14-digit) | `0001L016300001A` |

### Step 4 — Send the Request

**curl example:**
```bash
curl -X POST https://api.admin-link.net/v1/devices \
  -H "Content-Type: application/json" \
  -d '{
    "prdct": "NAS",
    "dev_reg_cd": "YOUR_REGISTRATION_CODE",
    "dev_id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    "dev_name": "My NAS Device",
    "serial_no": "01L016300001A",
    "remote_flg": "1",
    "proxy_flg": "0",
    "prdct_id": "L0163",
    "mac_adr": "AA:BB:CC:DD:EE:FF",
    "ms_mnemonic": "MNEMONIC_VALUE",
    "ms_ver": "1.0.0",
    "agt_mnemonic": "AGT_MNEMONIC",
    "agt_ver": "1.0.0",
    "os_cd": "LINUX",
    "ng_type": "001"
  }'
```

**Python example:**
```python
import requests
import json

url = "https://api.admin-link.net/v1/devices"
payload = {
    "prdct": "NAS",
    "dev_reg_cd": "YOUR_REGISTRATION_CODE",
    "dev_id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    "dev_name": "My NAS Device",
    "serial_no": "01L016300001A",
    "remote_flg": "1",
    "proxy_flg": "0",
    "prdct_id": "L0163",
    "mac_adr": "AA:BB:CC:DD:EE:FF",
    "ms_mnemonic": "MNEMONIC_VALUE",
    "ms_ver": "1.0.0",
    "agt_mnemonic": "AGT_MNEMONIC",
    "agt_ver": "1.0.0",
    "os_cd": "LINUX",
    "ng_type": "001"
}

response = requests.post(url, json=payload)

if response.status_code == 201:
    data = response.json()
    print("Registration successful")
    print(f"Device ID: {data['dev_id']}")
    print(f"Device ID changed: {data['dev_id_changed']}")
else:
    error = response.json()
    print(f"Error {response.status_code}: [{error['error_id']}] {error['error_msg']}")
```

### Step 5 — Handle the Response

#### Success (HTTP 201)

| Key | Required | Description |
|-----|----------|-------------|
| `dev_id` | ○ | Device ID (DB device ID if authenticated by MAC address) |
| `dev_id_changed` | ○ | `0` = unchanged, `1` = changed to Cloud-assigned ID |
| `agt_daily_sec` | ○ | Seconds offset for daily processing (integer 0–36000, for 10 hours). Daily processing runs at 22:00:00 + this value. Distributes daily processing across devices between 22:00 and 08:00 the next day. |
| `agt_upload_sec` | ○ | Seconds offset for agent info upload timing (integer 0–1800, for 30 minutes). Transmissions start at 0:00:00 + this value. **NAS**: status sent at 30-minute intervals. **Network device (AP/SW)**: status sent at 60-minute intervals. Connection client file is sent at user-specified intervals. |
| `tmp_reg_expiry` | — | Temporary registration expiry (UNIX timestamp). Only returned on `regist_cd` provisional registration. Expiry = registration time + 24 hours |

**dev_id_changed logic:**
```
If first install (provisional registration via regist_cd):
  → dev_id_changed = "0"  (device's own dev_id is returned as-is)

If device installed later (post-assignment registration):
  → dev_id_changed = "1"  (Cloud-assigned dev_id is returned)
```

#### Error (non-201)

```json
{
  "error_id": "4006",
  "error_msg": "フォーマットエラーが発生しました。",
  "error_field": "serial_no",
  "error_value": "invalid_serial"
}
```

### Step 6 — Diagnose Errors

#### HTTP 400 errors

| error_id | Meaning | Fix |
|----------|---------|-----|
| `4000` | Request BODY not specified. Key Required Error | Ensure Content-Type is application/json and body is not empty |
| `4001` | Required key missing. Key Required Error | Check all ○ required fields are present |
| `4002` | The request parameter item exists, but the value is not specified. Required error for value | Provide a value for all sent keys |
| `4004` | Value exceeds max digit count | Shorten the value to within the allowed length |
| `4005` | Invalid character type | Check field accepts the character set you are using |
| `4006` | Format error | Validate MAC address, IP address, or serial number format |
| `4007` | Value not in master (general) | Verify OS code, ng_type, ms_mnemonic, agt_mnemonic, prdct_id against master data |
| `4007` | Serial number master check error | Verify serial number format and that product code exists in master |
| `4008` | Azure agent not accepted | Agent version is outdated; update the agent software |
| `4009` | Forbidden characters | Remove half-width commas `,`, double quotes `"`, single quotes `'` from input |
| `4010` | dev_reg_cd authentication failed | Verify the device registration code is correct |

#### HTTP 401 errors

| error_id | Meaning | Fix |
|----------|---------|-----|
| `4010` | regist_cd authentication failed | Verify regist_cd exists in master |

#### HTTP 409 errors

| error_id | Meaning | Fix |
|----------|---------|-----|
| `4090` | Device already registered under another account | Check if device is registered to a different account |
| `4091` | Duplicate device detected | Two or more devices share the same dev_id or MAC address |

#### HTTP 500 errors

| error_id | Meaning |
|----------|---------|
| `5000` | General system error |
| `5001` | Failed to process new device registration |
| `5002` | Failed to update device info (DB update ①) |
| `5003` | Failed to update device info (DB update ②) |

## Business Rules Summary (always enforce)

### Mutual Exclusion
```
dev_reg_cd  XOR  regist_cd
  ↓                ↓
Manual mode     Auto mode
(Web UI)        (AP/SW only)
```
The unused key must be completely absent from the request body.

### Auto Registration (regist_cd) Constraints
| Key | Rule |
|-----|------|
| `dev_name` | Must NOT be sent |
| `serial_no` | Must NOT be sent |
| `remote_flg` | Must be `"1"` |
| `proxy_flg` | Must be `"0"` |

### Auto-trimmed Fields
Leading/trailing whitespace is stripped server-side for:
`dev_reg_cd`, `dev_name`, `serial_no`, `remarks`

### Duplicate Serial Number
Duplicate serial numbers are **not checked** — the DB allows duplicate serial numbers.

### Regist Code Format (when registered by ELECOM)
- Exactly **8 digits**
- Uniqueness guaranteed by combination of `regist_cd` + MAC address

### NextGiga (AP/SW) — Device ID Behavior
- **First install**: Device's own `dev_id` is used → returned as-is (`dev_id_changed=0`)
- **Post-assignment install**: Cloud-assigned `dev_id` takes precedence → returned with flag (`dev_id_changed=1`)

---

## Response Principles

1. Always reference specific key names and formats in answers
2. Never assume parameter values — prompt the user to confirm if uncertain
3. When diagnosing errors, map `error_id` to a cause and suggest a specific fix
4. Clearly distinguish behavior differences between **NAS** and **NextGiga (AP/SW)**
5. If the registration mode (manual vs auto) is unclear, ask before proceeding
